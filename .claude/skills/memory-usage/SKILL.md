# 記憶の活用（claude-memory MCP）

claude-memoryはプロジェクトの設計判断・知見・ユーザーの好みを蓄積し、セッションを跨いで再利用するためのツール。

## セッション開始時（必須）

セッション開始時に以下の検索を **必ず** 実行し、過去の文脈を把握する：

1. `memory_search` でプロジェクト名や現在の作業内容を検索
2. `memory_search` でユーザーの好みを検索（tags: preference）
3. 検索結果を踏まえて作業を開始する

## いつ記憶を検索するか

| 場面 | 検索クエリ例 |
|------|-------------|
| 設計判断をする前 | 関連する技術名・パターン名で検索し、過去の判断を確認 |
| バグに遭遇した時 | エラーメッセージ・モジュール名で検索し、既知の解決策を確認 |
| 新技術を導入する時 | 技術名で検索し、過去の選定理由・却下理由を確認 |
| Issue に着手する前 | 関連キーワードで検索し、過去の議論・合意を確認 |
| コードレビュー時 | 変更対象モジュール名で検索し、設計意図を確認 |

検索は **判断の前** に行う。判断してから「過去にどうだったか」を確認しても遅い。

## 保存ルール

### 保存する情報

以下はユーザーに確認せず `memory_save` で自動保存する：

| 保存する情報 | タグ | 具体例 |
|-------------|------|--------|
| 設計判断とその理由 | `design-decision` | 「RRFのk=60を選んだ理由」「drizzle-kit pushではなくgenerate+psqlにした理由」 |
| ユーザーの好み・作業スタイル | `preference` | 「ドキュメント鮮度を重視」「FigJamで日本語図」 |
| バグの原因と解決策 | `bug` | 「CIのflakyテストはタイムスタンプ未設定が原因」 |
| プロジェクト固有の制約 | `rule`, `architecture` | 「hoist=falseでdrizzle-kit pushが動かない」 |
| 議論の結論・合意事項 | `design-decision` | 「ドキュメントはWhy/What/Howの3分類で管理」 |
| 開発プロセスの知見 | `process` | 「worktreeが残るとknipがUnused filesを検出する」 |

### 保存しないもの

- 一般的な技術知識（公式ドキュメントに書いてあること）
- 一時的な作業状態（現在のブランチ、途中のデバッグ状況）
- コードそのもの（コードはGitで管理する）
- コードから読み取れる事実（関数名、ファイル構成など）

### 保存の判断基準

迷ったら「次のセッションで同じ状況に遭遇したとき、この情報があれば判断が速くなるか？」で考える。YESなら保存。

## scope の使い分け

| scope | 用途 | 例 |
|-------|------|-----|
| `project` | このプロジェクト固有の知識 | 「ハイブリッド検索のRRF k=60」 |
| `global` | プロジェクト横断で使える知見 | 「CI flakyテストの一般的な対処パターン」 |

デフォルトは `project`。明らかに他プロジェクトでも使える知見のみ `global` にする。

## タグ体系

保存時は必ず tags を付ける（検索精度に直結）：

- `design-decision` — 設計判断
- `preference` — ユーザーの好み
- `bug` — バグ原因・解決策
- `rule` — プロジェクトルール
- `architecture` — アーキテクチャ知識
- `process` — 開発プロセスの知見

複数タグを組み合わせてよい（例: `design-decision`, `architecture`）。

## 重複・矛盾する記憶の扱い

- 保存前に `memory_search` で類似の記憶がないか確認する
- 既存の記憶と同じ話題なら、新規保存ではなく既存を更新する（`memory_delete` → `memory_save`）
- 過去の判断を覆す場合は、古い記憶を削除してから新しい判断を保存する（理由も含めて）

## ADR連携

設計判断があった場合は以下の **両方** に記録する：

1. `docs/adr/` にADRファイルを作成（人間が読む構造化ドキュメント）
2. `memory_save` で記憶に保存（AIが検索するインデックス）

ADRは「なぜその判断をしたか」「何を検討して却下したか」を残す。
記憶には判断の要点とADRへの参照（`docs/adr/NNNN-*.md`）を含める。

## よくある間違い

| 間違い | 正しい対応 |
|--------|-----------|
| コードから読める事実（関数名・ファイル構成）を保存 | コードで十分。保存しない |
| タグ無しで保存 | 検索不能になる。必ず 1 つ以上タグを付ける |
| 類似記憶を確認せず重複保存 | 保存前に `memory_search` で類似を確認 |
| 過去判断を覆すとき古い記憶を残す | 古い記憶を削除してから新規保存 |
| グローバル知識を project スコープに保存 | 他プロジェクトでも使えるなら global |
| セッション開始時の検索をスキップ | 過去の文脈を逃す。必ず実行 |

## メンテナンス

- `memory_stats` で記憶の総数を定期確認
- `memory_list` で全記憶を確認し、古くなった情報がないかチェック
- 不要な記憶は `memory_delete` で削除
