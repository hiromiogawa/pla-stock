# 記憶の活用（harness 組み込みファイル memory）

本プロジェクトは **claude-memory MCP を使わない**。harness がシステムプロンプトで提供するファイルベース memory（プロジェクト memory ディレクトリの `MEMORY.md` インデックス + 型別エントリファイル）を使う。global `~/.claude/CLAUDE.md` の Memory セクション（MCP 強制）は本プロジェクトでは適用しない。

memory はプロジェクトの設計判断・知見・ユーザーの好みを蓄積し、セッションを跨いで再利用するための仕組み。実体は harness 提供の memory ディレクトリ（パスは harness が CWD から算出。skill 側でハードコードしない）。

## セッション開始時（必須）

セッション開始時に **必ず** `MEMORY.md` インデックスを読み、関連する型別エントリを辿って過去の文脈を把握する：

1. memory ディレクトリの `MEMORY.md`（索引）を読む
2. 現在の作業内容・ユーザーの好み（`feedback`/`user` 型）に関連するエントリを開く
3. 把握した文脈を踏まえて作業を開始する

## いつ記憶を参照するか

| 場面 | 参照方法 |
|------|-------------|
| 設計判断をする前 | MEMORY.md から関連技術・パターンのエントリを辿り過去の判断を確認 |
| バグに遭遇した時 | エラー・モジュール名で関連エントリ（既知の解決策）を確認 |
| 新技術を導入する時 | 技術名のエントリで過去の選定理由・却下理由を確認 |
| Issue に着手する前 | 関連キーワードのエントリで過去の議論・合意を確認 |
| コードレビュー時 | 変更対象モジュールのエントリで設計意図を確認 |

参照は **判断の前** に行う。

## 保存ルール

以下はユーザーに確認せず保存する（型別エントリを作成/更新し、MEMORY.md に索引追記）：

| 保存する情報 | 型 (metadata.type) | 具体例 |
|-------------|------|--------|
| ユーザーの好み・作業スタイル | `user` / `feedback` | 「ドキュメント鮮度を重視」「FigJam で日本語図」 |
| エージェントへの指示・訂正 | `feedback` | 「mutation は features/ に置く」「冗長な事前チェックは不要」 |
| プロジェクトの進行・制約・知見 | `project` | 「Phase C は read→mutation の順で移行」「seed は dev 専用」 |
| 外部参照ポインタ | `reference` | 「失敗履歴は docs/adr/0007」「rule-cycle 閾値基準は rule-cycle-meta」 |

設計判断は memory ではなく **ADR（`docs/adr/`）に一本化**する（design-decision skill 参照）。memory には設計判断本体を二重に持たない（必要なら `reference` 型で ADR へのポインタのみ）。

### 保存しないもの

- 一般的な技術知識（公式ドキュメント記載のこと）
- 一時的な作業状態（現在のブランチ、途中のデバッグ）
- コードそのもの / コードから読み取れる事実（関数名・ファイル構成）
- 設計判断の本体（ADR が SSoT）

### 保存の判断基準

「次セッションで同じ状況に遭遇したとき、この情報があれば判断が速くなるか？」 YES なら保存。

## エントリのフォーマット

harness memory 規約に従う（frontmatter `name`/`description`/`metadata.type` + 本文）。`MEMORY.md` は索引のみ（1 エントリ 1 行: `- [Title](file.md) — 要約`）。索引にエントリ本体を書かない。

## 重複・矛盾する記憶の扱い

- 保存前に MEMORY.md で類似エントリがないか確認する
- 同じ話題なら新規作成でなく既存エントリを更新する
- 過去の判断を覆す場合は古いエントリを書換/削除し、理由も残す

## よくある間違い

| 間違い | 正しい対応 |
|--------|-----------|
| claude-memory MCP（memory_search/save）を呼ぶ | 本プロジェクトは MCP 不使用。ファイル memory を使う |
| 設計判断を memory に保存 | ADR が SSoT。memory には持たない（必要なら reference ポインタのみ）|
| コードから読める事実を保存 | コードで十分。保存しない |
| MEMORY.md にエントリ本体を書く | 索引は 1 行要約のみ。本体は型別ファイル |
| 類似エントリを確認せず重複作成 | 保存前に MEMORY.md で確認 |
| セッション開始時の MEMORY.md 確認をスキップ | 過去文脈を逃す。必ず実行 |

## メンテナンス

- `MEMORY.md` を定期確認し、古い/不要なエントリを書換・削除する
- エントリ数が増えたら型別に整理する
